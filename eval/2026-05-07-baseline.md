## Encoder questions to add (output for the flywheel)

The primary deliverable of this evaluation is the set of questions the encoder should have asked. Each traces to a deficiency that earned a score of 3 or below in the original. Insert these directly into encoder-v1.md.

### Specificity
> Name three specific decisions, artifacts, or proper nouns that would appear ONLY in this brand's BCP and could not appear in a competitor's. Examples: a named partner (only if signed), a versioned spec choice, a license decision, a founder, a milestone date. If the BCP currently does not contain at least three such specifics across positioning, values, and representation, return to those sections and add them.

### Voice fidelity
> For every word in avoid_vocabulary, write a one-line context clause distinguishing the bad usage from any permissible usage. For every channel in channel_modulation, write at least one paired bad/good example labeled with the channel name. If the voice has tonal attributes, render each as an "X not Y" pair where Y is the closest plausible failure mode.

### Voice fidelity (machine readability)
> Are any avoid_vocabulary entries combining multiple words into a single compound entry? An agent doing word-by-word filtering needs each word independently flagged. Compound entries are acceptable only when the words share an identical rationale and never appear independently.

### Voice fidelity (channel register)
> For each channel, name who is reading at that surface and in what context. Surface alone (website, docs) is not enough — an agent benefits from knowing whether the reader is a journalist evaluating the brand, an engineer evaluating the spec, or a partner debugging an integration.

### Boundary coverage
> List the three most embarrassing plausible mistakes a content-generating agent could make about this brand — the three things a journalist, competitor, or aggressive sales rep could say that would be both believable and damaging. For each, write a hard_no or soft_no entry that would have prevented that mistake. Separately, name every direct competitor and write the rule for how to reference them.

### Claims rigor
> For every approved claim, name the external proof — a customer outcome with a metric, a signed partnership, a public artifact independent of the product itself, or a verifiable adoption number. If the only available proof is "we built the thing we are claiming we built," the evidence is circular and the claim is not yet verifiable; mark it requires_caveat or aspirational and write the verification trigger that would move it to verified.

### Claims rigor (aspirational vs signed) — highest-leverage question of this run
> Which claims about partners, customers, or adoption are aspirational versus signed? For each aspirational item, write the permissible state_as framing the agent may use today (typically "we are working toward..." or "we are in conversation with...") and the verification_trigger that promotes it to verified. Add a hard_no in boundaries.md against naming any party as partner/customer/adopter until the relationship is signed and the named party has approved being named publicly.

### Claims rigor (file separation)
> Are there biographical facts (founder, license, entity name) currently sitting in claims.md? Move them to brand.md frontmatter. Claims.md is for claims about value/capability/effect, not facts about the company.

### Representation accuracy
> For every direct named competitor and every plausible category-collapse description (e.g., "isn't this just X?"), write a never_compare_to entry that distinguishes this brand from that competitor or category in one line. For framing_traps, list the three most likely ways a journalist or analyst would mis-describe this brand if no correction were given, and write the one-sentence correction for each. Confirm that describe_as includes every distinctive descriptor present in internal strategy documents.

### Simplicity (cross-cutting)
> After every section is written, count the units (rules, entries, examples). For each unit, ask: does every word earn its place? If the same idea appears in two files, pick one home and cross-reference from the other. Cap each manifestation, why-clause, and example to one sentence unless the topic genuinely requires more.

### Simplicity (failure mode)
> Did simplicity cuts erase texture? Re-read each section and ask: was anything cut for visual symmetry rather than meaning? Was anything cut that was doing real work distinguishing this brand from a generic competitor in the same category? If yes, restore it.

## Texture repairs in v1.2.1 pass

The v1.2 → v1.2.1 pass identified that the simplicity criterion has a failure mode: cutting for symmetry rather than for meaning, and consolidating compound entries that hurt machine-readability for an agent-first BCP.

### voice.md repairs
- Restored "Optimistic, not utopian" and "Restrained, not aloof" attributes. Both were doing real work distinguishing Encoded from doom-merchants and from cold-bureaucrat tone. v1.2 cut them for visual symmetry (5 → 3), which was wrong.
- Re-split compound avoid entry ("supercharge / unleash / game-changer") into three independent entries. Compound entries hurt agent word-filtering for a human-readability win — wrong tradeoff for an agent-first BCP. Same fix applied to "AI takeover / future-proof" and "magic / seamless."
- Added "shows_up_in" register per channel naming who is reading at that surface and in what context. Surface alone is insufficient; an agent benefits from knowing whether the reader is a journalist, an engineer, or a partner debugging.

### values.md repairs
- Distinguished illustrative examples from documented decisions in "in practice" lines. The "auto-generate brand voice from a logo" example is now flagged as illustrative test rather than presented as a real past decision. Honest about what's lived versus what's hypothetical.
- Added "smallest piece" architectural specificity to Skeleton Key principle — names the actual design decision (granddaughter files, partial reads).
- Removed Zefr reference from value 3 example; generalized to "third-party classifiers" to stay clean of aspirational-partner naming.
